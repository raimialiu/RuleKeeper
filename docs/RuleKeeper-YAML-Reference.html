1. Configuration Overview

RuleKeeper uses YAML configuration files to define analysis policies. The configuration controls:

  • Which files to scan
  • Which rules to enable/disable
  • Severity levels and thresholds
  • Output format and visualization options
  • Custom rules and validators

2. File Locations

RuleKeeper searches for configuration files in the following order:

  1. rulekeeper.yaml
  2. .rulekeeper.yaml
  3. .rulekeeper.yml
  4. rulekeeper.yml

The search starts from the scan path and moves up through parent directories.

Tip: Use --config or -c to specify a custom configuration file path.

5. scan Section

scan:

  include:
    - "**/*.cs"
  exclude:
    - "**/obj/**"
    - "**/bin/**"
    - "**/*.Designer.cs"
    - "**/*.g.cs"
    - "**/*.generated.cs"
    - "**/Migrations/**"

  cache: true
Property Type Default Description
language string CSharp Programming language (currently only CSharp supported)
include array ["**/*.cs"] Glob patterns for files to include
exclude array [] Glob patterns for files to exclude
parallel boolean true Enable parallel file analysis
cache boolean true Enable analysis caching

Glob Pattern Examples

Pattern Matches
**/*.cs All C# files in any directory
src/**/*.cs All C# files under src directory
**/*Service.cs All files ending with Service.cs
**/Tests/** All files in any Tests directory

5.1 Baseline Configuration (Incremental Scanning)

New in v1.1: Baseline scanning allows you to add RuleKeeper to existing projects without being overwhelmed by violations in legacy code. Only new or changed code is reported. 
scan:
  # ... other scan options ...

  baseline:
    enabled: true
    mode: git
    git_ref: main
    include_uncommitted: true
    include_untracked: true
    filter_to_diff: true
    on_missing: warn
Property Type Default Description
enabled boolean false Enable baseline/incremental scanning
mode string git Baseline mode: git, file, or date
git_ref string HEAD~1 Git reference to compare against (branch, tag, or commit)
changed_lines_only boolean false Only scan changed lines (not entire changed files)
include_uncommitted boolean true Include uncommitted changes (staged and unstaged)
include_untracked boolean true Include untracked (new) files
filter_to_diff boolean true Filter violations to only those on changed lines
baseline_file string null Path to baseline file (when mode = file)
since_date string null Date to compare against (when mode = date, format: YYYY-MM-DD)
on_missing string warn Action when baseline is missing: warn, fail, ignore
auto_update boolean false Auto-update baseline file after scanning (file mode only)

Baseline Mode Examples

Git Mode - Compare against a branch or commit:

baseline:
  enabled: true
  mode: git
  git_ref: origin/main  # or: main, v1.0.0, abc123

File Mode - Compare against a stored baseline:

baseline:
  enabled: true
  mode: file
  baseline_file: .rulekeeper-baseline.json
  auto_update: true

Date Mode - Scan files modified since a date:

baseline:
  enabled: true
  mode: date
  since_date: "2026-01-01"

8. coding_standards Section

Supports two formats for defining rules:

Format 1: Simple List (Recommended)

coding_standards:
  naming:
    - id: CS-NAME-001
      name: "Class Naming Convention"
      description: "Classes must use PascalCase"
      severity: medium
      enabled: true
      pattern: "^[A-Z][a-zA-Z0-9]*$"
      message: "Class name must use PascalCase"
      fix_hint: "Rename to use PascalCase"
      applies_to:
        - classes

    - id: CS-NAME-007
      name: "Interface Naming"
      severity: medium
      pattern: "^I[A-Z][a-zA-Z0-9]*$"
      applies_to:
        - interfaces

Format 2: Full Format with Category Settings

coding_standards:
  naming:
    enabled: true
    severity: medium
    rules:
      - id: CS-NAME-001
        name: "Class Naming Convention"
        # ... rule properties

Rule Definition Properties

Property Type Required Description
id string Required Unique rule identifier
name string Optional Display name for the rule
description string Optional Detailed description
severity string Optional Severity level (default: medium)
enabled boolean Optional Enable/disable rule (default: true)
skip boolean Optional Skip rule (alias for enabled=false)
pattern string Optional Regex pattern for valid code
anti_pattern string Optional Regex pattern for invalid code
message string Optional Violation message
fix_hint string Optional Hint for fixing the violation
applies_to array Optional Syntax elements this rule applies to
file_pattern string Optional Limit rule to files matching pattern
exclude array Optional File patterns to exclude for this rule
parameters object Optional Additional parameters for the rule

applies_to Values

Value Description

structs Struct declarations

9. custom_validators Section

custom_validators:
  validate_password_complexity:
    type: regex
    pattern: "^(?=.*[a-z])(?=.*[A-Z])(?=.*\\d).{8,}$"
    message: "Password must be 8+ chars with upper, lower, and digit"

  validate_no_console_writeline:
    type: anti_pattern
    pattern: "Console\\.Write(Line)?"
    message: "Use structured logging instead of Console.Write"
    severity: medium
    exclude:
      - "**/Program.cs"
Property Type Required Description
type string Required Validator type: regex, anti_pattern, custom
pattern string Required* Regex pattern (*required for regex type)
message string Optional Violation message
severity string Optional Severity level
exclude array Optional File patterns to exclude

10. Advanced Custom Rules (NEW in v1.1)

New Feature: Define custom rules directly in YAML without writing C# code. Supports enhanced patterns, AST queries, multi-condition matching, expressions, and scripts.

10.1 Enhanced Pattern Rules

Enhanced patterns support captures, regex options, and message interpolation:

coding_standards:
  security:
    - id: CUSTOM-SEC-001
      name: "No Console Output"
      severity: medium
      anti_pattern_match:
        regex: "Console\\.(WriteLine|Write)\\s*\\("
        options: [ignorecase]
        message_template: "Use ILogger instead of Console.{match}"
      message: "Use ILogger instead of Console"
      exclude: ["**/Tests/**"]
Property Type Description
pattern_match / anti_pattern_match object Enhanced pattern configuration
.regex string The regex pattern
.options array Regex options: ignorecase, multiline, singleline, compiled
.captures object Named capture groups mapping
.scope string Scope: file, class, method, block
.message_template string Message with {capture} interpolation

10.2 AST Query Rules

Query AST nodes declaratively by kind, properties, and structure:

coding_standards:
  design:
    - id: CUSTOM-DESIGN-001
      name: "No Public Fields"
      severity: high
      ast_query:
        node_kinds: [FieldDeclaration]
        properties:
          IsPublic: true
          IsConst: false
        no_attribute: [Obsolete]
      message: "Public fields should be properties"
Property Type Description
node_kinds array AST node kinds to match (MethodDeclaration, ClassDeclaration, etc.)
properties object Property values to match (IsPublic: true, etc.)
has_children array Required child node kinds
no_children array Excluded child node kinds
has_ancestor array Required ancestor node kinds
no_ancestor array Excluded ancestor node kinds
text_matches string Regex to match node text
name_matches string Regex to match node name/identifier
has_attribute / no_attribute array Required/excluded attributes
languages array Limit to specific languages

10.3 Multi-Pattern Rules (AND/OR/NOT)

Combine multiple conditions with logical operators:

coding_standards:
  security:
    - id: CUSTOM-SEC-002
      name: "SQL Injection Risk"
      severity: critical
      match:
        all:  # AND: all conditions must match
          - pattern:
              regex: "(ExecuteSqlRaw|FromSqlRaw)"
          - ast_query:
              has_children: [BinaryExpression]
        none:  # NOT: none of these should match
          - pattern:
              regex: "@\\w+"
      message: "Use parameterized queries"
Property Type Description
all array All conditions must match (AND logic)
any array At least one condition must match (OR logic)
none array None of these conditions should match (NOT logic)
min_any_matches integer Minimum matches required for 'any' (default: 1)

10.4 Expression Rules

Use C# expressions for complex validation logic:

coding_standards:
  design:
    - id: CUSTOM-DESIGN-002
      name: "Method Too Long"
      severity: medium
      expression:
        condition: "Node is IMethodNode m && GetLineCount() > (int)Parameters[\"max_lines\"]"
        message_expression: "$\"Method has {GetLineCount()} lines (max: {Parameters[\"max_lines\"]})\""
      parameters:
        max_lines: 30

10.5 Script Rules

Full C# scripts for the most complex validation scenarios:

coding_standards:
  custom:
    - id: CUSTOM-SCRIPT-001
      name: "Custom Complexity Check"
      severity: high
      script:
        code: |
          var violations = new List<Violation>();
          foreach (var method in GetMethods())
          {
              var complexity = CalculateComplexity(method);
              if (complexity > GetParameter<int>("max_complexity", 10))
                  violations.Add(CreateViolation(method, $"Complexity: {complexity}"));
          }
          return violations;
      parameters:
        max_complexity: 10

custom_rules:
  - path: "./custom-rules/MyCompany.Rules.dll"

  - nuget: "MyCompany.RuleKeeper.Rules:1.0.0"
Property Type Description
path string Path to custom rule assembly DLL

Note: Custom rule assemblies can also contain fix providers. Any class implementing IFixProvider with the [FixProvider] attribute will be automatically discovered.

10.1 custom_fix_providers Section

Specifies assemblies containing custom fix providers for the fix command. This is optional - fix providers in custom_rules assemblies are also loaded automatically.

custom_fix_providers:
  - path: "./custom-fixes/MyCompany.Fixes.dll"

  - path: "./plugins/AnotherFixer.dll"
Property Type Description
path string Path to custom fix provider assembly DLL

Creating Custom Fix Providers

To create a custom fix provider, reference the RuleKeeper.Sdk package and implement IFixProvider:

// MyCustomFixProvider.cs
using RuleKeeper.Sdk;
using RuleKeeper.Sdk.Attributes;

[FixProvider("My Custom Fixes", Category = "Custom")]
public class MyCustomFixProvider : BaseFixProvider
{
    public override IEnumerable<string> SupportedRuleIds =>
        new[] { "CUSTOM-001", "CUSTOM-002" };

    public override IEnumerable<CodeFix> GetFixes(
        Violation violation, string sourceCode)
    {
        // Your fix logic here
        yield return CodeFix.CreateReplacement(
            violation,
            "Fix description",
            startColumn: 1,
            endColumn: 10,
            "original",
            "replacement"
        );
    }
}
Tip: Use rulekeeper fix --list-fixable --verbose to see all registered fix providers including custom ones.

12. Severity Levels

Level Value Description Typical Use
Critical critical Most severe - immediate action required Security vulnerabilities, blocking async
High high High impact - should be fixed soon Empty catch blocks, async void
Medium medium Medium impact - fix when possible Naming conventions, method length
Low low Low impact - minor improvements Private field naming, async suffix
Info info Informational - suggestions only Style suggestions, recommendations
Best Practice: Start with pre-built policies enabled, then customize severity levels and add custom rules as needed for your organization.
