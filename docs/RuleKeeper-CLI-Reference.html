3. scan - Analyze Code

rulekeeper scan <path> [options]

The primary command for analyzing C# code against configured policies.

Arguments

Argument Description Default
<path> Path to scan (file, directory, project, or solution) . (current directory)

Configuration Options

Option Alias Description Default
--config -c Path to configuration file Auto-discover rulekeeper.yaml
--language -l Programming language (currently only CSharp) CSharp

Output Options

Option Alias Description Default
--output -o Primary output format: console, json, sarif, html console
--format -F Additional output formats to generate (can specify multiple) None
--output-file Output file path for reports None
--summary-only --summary Show only summary tables without violation details false
--no-viz --no-visualization Disable visualization charts in reports false
--no-table Disable tabular summary in console output false
--no-color Disable colored output false
--verbose -v Enable verbose output false

Severity & Filtering Options

Option Alias Description Default
--severity -s Minimum severity to report: Info, Low, Medium, High, Critical Info
--fail-on -f Severity level that causes non-zero exit code None
--include -i File patterns to include (glob) **/*.cs
--exclude -e File patterns to exclude (glob) None

Threshold Options

Option Alias Description Default
--threshold-percent --tp Maximum allowed violation percentage (0-100) 0
--critical-threshold --ct Maximum allowed critical violations 0
--high-threshold --ht Maximum allowed high severity violations 0
--total-threshold --tt Maximum allowed total violations 0
Note: Threshold value of 0 means any violation of that type causes failure (when --fail-on is set). Set to a positive number to allow some violations.

Performance Options

Option Alias Description Default
--parallel -p Enable parallel analysis true
--no-cache Disable caching false

4. fix - Auto-Fix Violations NEW

rulekeeper fix <path> [options]

Automatically fixes code violations that have available auto-fix providers. Supports dry-run mode to preview changes before applying them.

Arguments

Argument Description Default
<path> Path to the file, directory, or project to fix . (current directory)

Options

Option Alias Description Default
--config -c Path to RuleKeeper configuration file Auto-discover
--rule -r Fix only violations of a specific rule ID (e.g., CS-NAME-001) All fixable rules
--category -t Fix only violations in a specific category (e.g., naming, async, sec, exc) All categories
--dry-run -d Preview fixes without applying them false
--no-backup Don't create backup files (.bak) before fixing false
--interactive -i Confirm each fix before applying false
--list-fixable -l List all fixable rule IDs and exit false
--verbose -v Show detailed output false

Supported Auto-Fix Rules

Rule ID Category Fix Description
CS-NAME-001 Naming Class naming to PascalCase
CS-NAME-002 Naming Method naming to PascalCase
CS-NAME-003 Naming Add 'Async' suffix to async methods
CS-NAME-004 Naming Private field naming to _camelCase
CS-NAME-005 Naming Constant naming to PascalCase
CS-NAME-006 Naming Property naming to PascalCase
CS-NAME-007 Naming Add 'I' prefix to interfaces
CS-NAME-008 Naming Parameter naming to camelCase
CS-NAME-009 Naming Local variable naming to camelCase
CS-NAME-010 Naming Add 'T' prefix to type parameters
CS-ASYNC-002 Async Replace .Result/.Wait() with await
CS-ASYNC-003 Async Add ConfigureAwait(false)
CS-EXC-001 Exceptions Add TODO comment for empty catch blocks
CS-EXC-004 Exceptions Replace 'throw ex;' with 'throw;' to preserve stack trace

Custom Fix Providers

You can create and use custom fix providers to handle fixes for custom rules or override built-in fix behavior.

Using Custom Fix Providers

Add custom fix provider assemblies in your rulekeeper.yaml:

# Load from custom_fix_providers (dedicated fix providers)
custom_fix_providers:
  - path: "./my-fixes/MyCompany.Fixes.dll"

# Or, fix providers in custom rule assemblies are also loaded
custom_rules:
  - path: "./my-rules/MyCompany.Rules.dll"  # Can also contain fix providers

Creating a Custom Fix Provider

Implement the IFixProvider interface from the SDK:

using RuleKeeper.Sdk;
using RuleKeeper.Sdk.Attributes;

[FixProvider("My Custom Fix Provider", Category = "Custom")]
public class MyCustomFixProvider : BaseFixProvider
{
    public override IEnumerable<string> SupportedRuleIds => new[]
    {
        "CUSTOM-001",
        "CUSTOM-002"
    };

    public override IEnumerable<CodeFix> GetFixes(Violation violation, string sourceCode)
    {
        // Implement your fix logic
        var line = GetLine(sourceCode, violation.StartLine);
        if (line == null) yield break;

        yield return CodeFix.CreateReplacement(
            violation,
            description: "Fix custom violation",
            startColumn: violation.StartColumn,
            endColumn: violation.EndColumn,
            originalText: "bad",
            replacementText: "good"
        );
    }
}

Viewing All Fix Providers

# List all fixable rules including custom providers
rulekeeper fix --list-fixable --verbose
Tip: Always use --dry-run first to preview changes before applying them to your codebase.

10. generate-analyzer-config - Generate IDE Analyzer Configs NEW

rulekeeper generate-analyzer-config [options]

Alias: rulekeeper gen-config

Generates IDE analyzer configuration files that developers can add to their projects for real-time enforcement in Visual Studio, VS Code, and Rider.

Options

Option Alias Description Default
--format -f Output format: globalconfig, ruleset, props, or all globalconfig
--output -o Output directory Current directory
--config -c Path to RuleKeeper configuration file Auto-discover
--strict Use error severity for all enabled rules false
--force Overwrite existing files without prompting false

Generated Files

File Purpose IDE Support
.globalconfig Modern Roslyn analyzer configuration (.NET 5+) Visual Studio 2022+, Rider 2023+
RuleKeeper.ruleset Legacy Roslyn analyzer ruleset Visual Studio 2019+
Directory.Build.props MSBuild integration with analyzer packages All MSBuild-based IDEs
Tip: Use --format all to generate all configuration files at once, then choose the one that works best for your environment.

12. Usage Examples

Basic Scan

# Scan current directory
rulekeeper scan

# Scan specific path
rulekeeper scan ./src

# Scan a solution file
rulekeeper scan ./MyProject.sln

Auto-Fix Violations

# List all auto-fixable rules
rulekeeper fix --list-fixable

# Preview fixes without applying (dry run)
rulekeeper fix ./src --dry-run

# Fix all violations
rulekeeper fix ./src

# Fix specific rule only
rulekeeper fix ./src --rule CS-NAME-003

# Fix naming issues only
rulekeeper fix ./src --category name

# Interactive mode - confirm each fix
rulekeeper fix ./src --interactive

# Fix without creating backups
rulekeeper fix ./src --no-backup

Output Formats

# JSON output to file
rulekeeper scan ./src -o json --output-file report.json

# SARIF for GitHub Code Scanning
rulekeeper scan ./src -o sarif --output-file results.sarif.json

# HTML report
rulekeeper scan ./src -o html --output-file report.html

# Multiple formats at once
rulekeeper scan ./src -o console -F json -F sarif

Summary Mode

# Show only summary tables (great for CI/CD)
rulekeeper scan ./src --summary-only

# Clean CI output
rulekeeper scan ./src --summary-only --no-color

Threshold Configuration

# Fail if more than 5 critical violations
rulekeeper scan ./src --fail-on Critical --ct 5

# Allow up to 10% violation rate
rulekeeper scan ./src --tp 10

# Allow 50 total violations
rulekeeper scan ./src --fail-on High --tt 50

# Strict mode - fail on any high or above
rulekeeper scan ./src --fail-on High --ht 0 --ct 0

Generate IDE Configurations

# Generate .globalconfig (recommended)
rulekeeper generate-analyzer-config

# Generate all configuration files
rulekeeper generate-analyzer-config --format all --output ./my-solution

# Strict mode (all rules as errors)
rulekeeper generate-analyzer-config --format all --strict

# Generate EditorConfig
rulekeeper generate-editorconfig

File Filtering

# Scan only service files
rulekeeper scan ./src -i "**/*Service.cs"

# Exclude test files
rulekeeper scan ./src -e "**/Tests/**" -e "**/*Tests.cs"

# Combined filtering
rulekeeper scan ./src -i "**/*.cs" -e "**/obj/**" -e "**/bin/**"

CI/CD Integration

# GitHub Actions / Azure DevOps
rulekeeper scan ./src \
    -o sarif \
    --output-file results.sarif.json \
    --fail-on High \
    --ct 0 \
    --no-color \
    --summary-only

# Auto-fix in CI (with verification)
rulekeeper fix ./src --no-backup
rulekeeper scan ./src --fail-on High

# Verbose debug mode
rulekeeper scan ./src -v --no-cache
Tip: Use --summary-only --no-color for cleaner CI/CD logs while still getting tabular summaries.
